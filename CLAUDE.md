# 项目概述
- Rust 语言实现常用工具集
- 工具以插件的形式实现，有统一的入口，聚合管理，插件可用快捷方式唤出
- GUI 形式，布局类似微软 PowerToys，图形库使用 egui

# 插件列表
- 密码管理工具，针对不同网站的账号密码存储记录
- hosts 管理工具，新增不同环境 hosts，启用禁用更环境 hosts
- JSON 编辑器，格式化处理，JSON 字符串解析；压缩、转义 JSON
- diff 文本代码对比工具
- 轻量级 Postman HTTP API 调试工具
- 其他插件后续补充添加

# 本地数据存储
- SQLite 优先

# 目录与代码规则
- 新增的代码按现有代码风格编写，如方法的拆分逻辑、方法名称及变量名称等与当前的命名规则保持一致
- 新增目录与当前目录风格一致，包括目录创建意图和命名规则
- 新增公共 API 必须包含文档注释（`///`）
- 所有 use 语句统一放在最前面，不要放在代码中间或后面
- 禁止将代码块如 {} 内的内容写成一行的写法，遵循 cargo fmt 格式规范

# 代码提交
- **代码提交前需要让我确认通过后才可以提交**
- 简要总结提交内容，包括涉及的功能模块、新增、修改、删除的代码文件数量、完成目标等，方便我审核确认，修改的内容可以以 diff 的方式在编辑器上查看
- 测试验证通过后及 code review 完成后才开始代码提交流程
- git commit 信息要符合本次修改目标，使用简体中文，80个中文字符以内

# 代码自动化审核规则
- 每当完成一个编码任务（例如新增功能、修复 Bug、重构）并且修改了至少一个源代码文件后，**必须**调用 `code-reviewer` 子代理对本次变更进行审查。
- 调用前，先通过 `git diff` 获取变更内容，并将变更内容和本次任务的目标（如“修复了登录失败无提示的 Bug”）一并传递给 `code-reviewer`。
- 审查完毕后，将子代理的审查报告直接输出给我，不需要再做额外总结。
- 如果我明确说“跳过审查”，则本次可以不调用。

# Code Review 规则
- 库代码禁止使用 `unwrap()` / `expect()`；必须返回 `Result` 或使用 `thiserror`/`anyhow` 适当处理错误。
- 禁止使用 `as` 进行可能丢失数据的强制类型转换；使用 `From`/`Into`/`TryFrom` 替代。
- 每个 `unsafe` 块上方必须有 `// SAFETY:` 注释，明确解释其安全条件如何被满足。
- 优先使用 `&str`、`&[T]`、`&Path` 等宽引用作为函数参数，而非对应的自有类型。
- 避免在热路径上调用 `clone()`，优先考虑借用或使用 `Cow`。
- 所有数据库操作或 I/O 操作必须在 `async` 函数中执行，并使用 `tokio::time::timeout` 或类似机制设置超时。
- 修改全局可变状态时（如配置文件、环境变量），必须记录 `info!` 级别的日志。
- 变更涉及重构时，需保证原有单元测试全部通过，且为新逻辑补充测试。
